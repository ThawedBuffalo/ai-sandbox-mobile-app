## Approval Governance Workflow

This repository uses a dual-tracking approval model: documentation plus Jira.

- Workflow definition: `60-approvals/templates/APPROVAL-WORKFLOW.md`
- Approval decision register: `60-approvals/records/APPROVAL-DECISIONS.md`
- Architecture leadership checklist: `60-approvals/templates/ARCHITECTURE-REVIEW-CHECKLIST.md`
- Security leadership checklist: `60-approvals/templates/SECURITY-REVIEW-CHECKLIST.md`
- Compliance review template: `60-approvals/templates/COMPLIANCE-REVIEW-TEMPLATE.md`

Default review behavior is parallel and agile:
- Architecture leadership review and security leadership review run in parallel.
- Compliance review runs in parallel when needed, but can block release promotion based on environment rules.
- Reviewers can use `Changes Requested` to require artifact updates before final approval; each round must be traceable in Jira and the decision register.

Environment gating:
- Development, Staging, UAT: `Approved` or `Approved with conditions` (with owner and due date)
- Production: `Approved` by default; conditional promotion requires formal exception
- Any environment: promotion is blocked while a required approval issue is `Changes Requested`

## Local Setup

### 1. Project Configuration

Copy the config template and fill in your project-specific tool URLs and keys:

```bash
cp project-config.template.json project-config.json
```

Edit `project-config.json` with your values:

| Field | Description |
|---|---|
| `jira.projectKey` | Jira project key (e.g. `MYPROJECT`) |
| `jira.issueTypeReview` | Issue type for review tasks (default: `Task`) |
| `jira.approvalWorkflow.issueType` | Jira issue type used by approval-gate script |
| `jira.approvalWorkflow.labels` | Labels applied to all approval-gate review issues |
| `jira.approvalWorkflow.domainLabels` | Per-domain labels for architecture/security/compliance reviews |
| `jira.approvalWorkflow.transitionByDecision` | Decision-to-transition mapping used to mirror workflow states in Jira |
| `confluence.spaceKey` | Confluence space key |
| `confluence.parentPageId` | Parent page ID for publishing |
| `confluence.parentPageUrl` | Full URL to the Confluence parent page |
| `miro.boardUrl` | Miro board URL for architecture diagrams |
| `miro.boardId` | Miro board ID (from the URL) |
| `figma.fileUrl` | Figma file URL for UX design |
| `figma.fileKey` | Figma file key (from the URL) |
| `project.name` | Feature/initiative name |
| `project.description` | One-line description |

### 2. Environment Variables (Secrets)

Create your local `.env` from template:

```bash
cp .env.template .env
```

Required variables:

- `JIRA_HOST` — Atlassian instance URL (e.g. `https://<tenant>.atlassian.net`)
- `JIRA_EMAIL` — Jira user email for API auth
- `JIRA_API_TOKEN` — Jira API token
- `JIRA_CA_CERT_PATH` (optional) — path to PEM root/intermediate CA if your Jira TLS chain is corporate/self-signed
- `JIRA_TLS_INSECURE` (optional, default `false`) — set `true` only for temporary local troubleshooting

> **Note:** Secrets (API tokens) live in `.env`. Non-secret project config (project keys, URLs, page IDs) lives in `project-config.json`.

### 3. Install Dependencies

```bash
cd .github/skills/jira-mcp-server && npm install && cd ../../..
```

The MCP server starts automatically via VS Code MCP registration (`.vscode/mcp.json`).
To run manually:


Server entrypoint:

MCP registration:
- `.vscode/mcp.json`

## Confluence Publishing

Publish Markdown docs to Confluence using:

- `.github/scripts/publish-confluence-add.py`

Example:

```bash
python3 .github/scripts/publish-confluence-add.py \
  --title "SSD - My Feature" \
  --space-key "<from project-config.json confluence.spaceKey>" \
  --parent-id "<from project-config.json confluence.parentPageId>" \
  --md-path "50-security/SSD-my-feature.md"
```

The script also reads defaults from `project-config.json` when `--space-key` or `--parent-id` are omitted.

## Jira Approval Workflow Automation

Use `.github/scripts/jira-approval-review.js` to create or update leadership approval issues and map decisions to Jira transitions.

Create architecture review issue:

```bash
node .github/scripts/jira-approval-review.js \
  --mode create \
  --domain architecture \
  --summary "Architecture Leadership Approval - ADD v1.2" \
  --artifact "40-architecture/ADD.md" \
  --checklist "60-approvals/templates/ARCHITECTURE-REVIEW-CHECKLIST.md" \
  --target-env development \
  --decision "In Review"
```

Update existing review issue to a decision:

```bash
node .github/scripts/jira-approval-review.js \
  --mode update \
  --issue-key CXINIT2-2740 \
  --domain security \
  --decision "Approved with conditions" \
  --target-env uat \
  --conditions "R4 audit evidence must be attached" \
  --due-date 2026-03-20
```

Mark review issue as Changes Requested:

```bash
node .github/scripts/jira-approval-review.js \
  --mode update \
  --issue-key CXINIT2-2740 \
  --domain security \
  --decision "Changes Requested" \
  --target-env development \
  --conditions "SEC-CR-001: add R4 logging audit evidence link" \
  --due-date 2026-03-20
```

After artifact updates are submitted, return issue to In Review:

```bash
node .github/scripts/jira-approval-review.js \
  --mode update \
  --issue-key CXINIT2-2740 \
  --domain security \
  --decision "In Review" \
  --target-env development
```

### Jira TLS Troubleshooting

If a Jira command fails with `self-signed certificate in certificate chain`:

1. Prefer secure fix: set `JIRA_CA_CERT_PATH` in `.env` to your corporate/root CA PEM file.
2. Retry the command.
3. Use `JIRA_TLS_INSECURE=true` only as a temporary local workaround.

## Security and Secrets

- Do **not** commit `.env` or credentials.
- Use `.env.template` for safe sharing.
- `node_modules/` is ignored and not tracked.
